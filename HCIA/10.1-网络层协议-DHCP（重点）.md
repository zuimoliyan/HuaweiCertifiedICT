## 一、DHCP
### 1.产生背景
1. 当网络规模比较大的时候，给终端配置IP地址比较繁琐
2. 可能手工配置IP地址会导致IP地址冲突
3. 可以让IP地址的利用率更高

### 2.DHCP的报文封装
DHCP封装在UDP之上，目的端口是`67`，源端口是`68`

### 3.<font color="#ff0000">DHCP的报文和工作原理</font>
#### 1.DHCP的报文
1. <font color="#ff0000">DHCP discover--发现报文</font>：终端发送DHCP discover用于寻找DHCP服务器。该报文是==广播==，源IP是`0.0.0.0`，目的IP是`255.255.255.255`
2. <font color="#ff0000">offer--通告报文</font>：用于DHCP服务器发送分配给终端的**IP地址、DNS信息、GW信息、掩码信息**等等。该报文是==单播==，源地址是DHCP服务器的地址，目的地址是分配给终端的IP地址
3. <font color="#ff0000">request报文</font>：用于回应DHCP服务器的offer报文，确认**使用**哪台DHCP服务器分配的的IP地址。该报文是==广播==，源IP是`0.0.0.0`，目的IP是`255.255.255.255`
4. <font color="#ff0000">ACK报文</font>：用于DHCP服务器回应终端，表示确认。该报文是==单播==，源地址是DHCP服务器的地址，目的地址是分配给终端的IP地址
5. Release：终端用于通过DHCP获取IP地址。该报文是==单播==报文，源IP是自己获取的IP地址，目的IP就是服务器的IP地址
6. Decline：当获取的地址和线网中的IP地址冲突，那么获取IP地址的终端会发出Decline报文。该报文是==广播==报文，源地址是`0.0.0.0`，目的IP是`255.255.255.255`
为什么 DHCP 是一个有状态的协议？
	因为 DHCP 会记录获取 IP 的终端的 MAC 还会记录租约信息，以及获取地址的状态（冲突、使用、超时、空闲）
#### 2.工作原理

### 3.DHCP的配置
#### 1.基于接口
<font color="#ff0000">终端和 DHCP 服务器必须在同一个网段</font>
1. 全局开启`DHCP enable`
2. 接口配置
```
int g 0/0/0
 ip address <IP地址> <子网掩码>
 dhcp select interface
 dhcp server excluded-ip-address <IP地址> //排除某个IP地址，如果是 <IP地址1> <IP地址2>则是从<IP地址1>排除到<IP地址2>
```

#### 2. 基于全局（实际用的多）
<font color="#ff0000">终端和 DHCP 服务器可以在同一个网段，也可以在不同的网段</font>
1. 配置 IP 地址池
   ```
	ip pool <地址池名称>
	 gateway-list <IP地址>
	 network <IP网段> mask <子网掩码>
	 dns-list <DNS>
	 ```
2. 接口下配置基于全局
   int g 0/0/0
    ip address <IP地址> <子网掩码>
    dhcp select global
   ```
注意：分配 IP 地址的顺序：
- 静态绑定
- Idle --->从大到小分（有些设备分配 IP 的时候是<font color="#ffc000">随机</font>的）
- Expired
- Conflict（冲突）

#### 案例 2：终端在双 DHCP 服务器环境下的选择规则
![](DICM/Pasted%20image%2020251123151724.png)
##### 分析：
![](DICM/Pasted%20image%2020251123151646.png)
1. 首先，终端发送 `DHCP Discover` 请求IP 地址
2. 此时 `AR1` 和 `AR2` 都会回复 `DHCP offer`，但终端会<font color="#ff0000">按发送的时间顺序选择</font>（<font color="#6425d0">选择早发送的那个</font>），即 `192.168.1.254` 发送的
3. 终端回复 `DHCP Request` 报文表示接收此IP 地址
4. `AR1` 回复 `DHCP ACK` 表示收到
5. 最后终端发送三个（有些是五个）免费ARP 来判断此IP是否已经被使用
#### 案例 3：终端请求到路由器 1 的DHCP 后，在续约时路由器 1 下线，路由器 2 的操作
由于模拟器中PC 不能发出续租报文，<font color="#ff0000">所以使用路由器来模拟</font>
![](DICM/Pasted%20image%2020251123154621.png)
##### 分析
###### 第一部分：到期正常续租
![](DICM/Pasted%20image%2020251123154457.png)
1. 正常获取DHCP
2. <font color="#ffc000">租期达到一半时</font>，终端向DHCP 服务器发出续租，DHCP 服务器回复收到
3. 第二次续租，与 `2` 一致
###### 第二部分：到期未能正常续租
![](DICM/Pasted%20image%2020251123154555.png)
1. 到达租期一半后，终端向DHCP 服务器发出续租申请，未得到回应
	按照**指数退避机制**：
	- 租约到期前 **T/4**：客户端重新发送续租申请，若仍未响应，继续等待至下一阶段
	- 租约到期前 **T/8**：客户端重新发送续租申请
	- ..........
	若租约到期前仍未收到任何 DHCP 服务器回应客户端<font color="#ff0000">释放当前 IP</font>，进入重新获取流程
2. 重新进入DHCP 的获取流程

#### 案例 4：单DHCP但是地址冲突
![](DICM/Pasted%20image%2020251123160017.png)
##### 分析

![](DICM/Pasted%20image%2020251123155952.png)
1. 正常请求DHCP 服务器获取IP 地址
2. 获取到之后，发送免费ARP，看是否已经被使用，<font color="#8064a2">此时发现有终端回复（即已经被使用了）</font>
3. 发送 `DHCP Decline` 给DHCP 服务器，表示地址冲突，并重新执行DHCP 获取进程

此时，在DHCP 服务器上，<font color="#ff0000">该冲突地址会被标记成冲突</font>

## 三、相关命令
1. `dhcp enable`：系统视图下使能 DHCP 服务
2. `dhcp server lease { day day [ hour hour [ minute minute ] ]  unlimited }`：配置 IP 地址租期，缺省情况下，IP 地址的租期为 1天
3. `dhcp server excluded-ip-address start-ip-address[ end-ip-address ]`：配置地址池中不参与自动分配的工P 地址保留以分配给其他的服务
4. `dhcp server static-bind ip-address ip-address mac-address mac-address`：采用静态地址绑定方式将接口地址池中的 IP 地址与 MAC 地址绑定。
5. `display ip pool name <地址池名称> all`：查看该地址池的全部配置（含地址范围、DNS、租期配置等）+ 已分配 IP
